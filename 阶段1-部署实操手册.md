## 0. 部署前审计（先看，再动）

我看了你的两个文件，有 4 个点建议先处理：

### A. ⚠️ debian1 内存极度紧张

debian1 上同时跑 PD + TiKV + TiDB + Prometheus + Grafana + Alertmanager，1GB 内存下：
- TiDB 进程软上限已设 `performance.server-memory-quota: 512MB`
- TiKV `storage.block-cache.capacity: 150MB` + write-buffer ~128MB ≈ 280MB
- PD 常驻 ~80–120MB
- Prometheus + Grafana + Alertmanager ≥ 250MB

加起来 ~1.1GB+，会非常贴近 OOM 边界。**任何稍大的 SQL、Region scan 或 ANALYZE 都会被 OOM-kill**。

建议（任选一个）：
1. **可行性优先**：把 Grafana / Alertmanager 挪到 debian2 或 debian3 单独承载（修改 `grafana_servers` / `alertmanager_servers` 的 host）。
2. **资源优先**：给 debian1 加内存到 2GB（如果是 VirtualBox，关机 → 设置 → 系统 → 4GB 之类）。
3. **PoC 接受 OOM**：保持现状，但接受跑实验时 debian1 会偶尔 OOM、需要重启 TiDB / TiKV。如果只是验证部署流程，这条可以接受。

### B. `prep_tidb.sh` 缺 `nproc` limit

脚本里只设了 `nofile` 和 `stack`，没有设 `nproc`。TiKV 在高并发时如果 `nproc` 是默认 1024 会触发 `fork: Resource temporarily unavailable`。建议补：

```bash
# /etc/security/limits.d/99-tidb.conf 追加
tidb soft nproc 8192
tidb hard nproc 8192
```

### C. `prep_tidb.sh` 没自动写入控制机的公钥

脚本只建了 `tidb` 用户和空的 `authorized_keys`，控制机（macOS）的公钥还是要手动 `ssh-copy-id` 或 scp 进去。这一步在「步骤 2」里补上。

### D. raft 心跳调慢了，要意识到副作用

`tidb-topology.yaml` 里：
```yaml
raftstore.raft-base-tick-interval: "2s"
raftstore.raft-heartbeat-ticks: 6
raftstore.raft-election-timeout-ticks: 50
```
- heartbeat = 2s × 6 = 12s
- election = 2s × 50 = 100s

这是为了省 CPU 做的极限裁剪，**副作用是**：一旦某个 TiKV 失联，要 100s 才会触发重新选举。EX5（节点重启验证自动恢复）的时候会看到恢复时间比文档默认值（10s 级）慢很多 —— 不是 bug，是配置如此。

## 步骤 1：控制机准备 + 安装 TiUP

> 在你的 macOS 控制端执行。预计 5 min。

```bash
# 安装 tiup（如果还没装）
curl --proto '=https' --tlsv1.2 -sSf https://tiup-mirrors.pingcap.com/install.sh | sh
source ~/.zshrc   # 或 ~/.bash_profile

# 验证
tiup --version

# 安装 cluster 组件
tiup install cluster
tiup cluster --version

# 生成给 tidb 用户用的 SSH key（如果还没有）
ssh-keygen -t rsa -b 4096 -f ~/.ssh/id_rsa_tidb -N "" -C "tidb-deploy@$(hostname)"
ls -l ~/.ssh/id_rsa_tidb*
```

**对照 KP76 / KP77**：观察 `~/.tiup/` 目录结构（components、storage、profile），理解 TiUP 是一个版本管理 + 部署编排工具，所有集群元数据都存在控制机本地的 `~/.tiup/storage/cluster/clusters/<name>/meta.yaml`。

## 步骤 2：三台 VM 跑 prep_tidb.sh

> 三台 Debian VM 都要跑，**用 root**。预计 5 min（每台）。

```bash
# 以 debian1 为例，重复 3 次
ssh root@192.168.56.11 'bash -s' < install/prep_tidb.sh
ssh root@192.168.56.12 'bash -s' < install/prep_tidb.sh
ssh root@192.168.56.13 'bash -s' < install/prep_tidb.sh
```

（如果是先用 vagrant / sudo 用户登录，先 `sudo -i` 再粘贴脚本也行。）

**审计 B 的补丁**（追加 nproc limit），任选一处实施：

- 在 `prep_tidb.sh` 的 limits 段里追加 `tidb soft/hard nproc 8192` 两行后重跑；
- 或直接在三台 VM 上：

```bash
for ip in 192.168.56.11 192.168.56.12 192.168.56.13; do
  ssh root@$ip 'cat >>/etc/security/limits.d/99-tidb.conf <<EOF
tidb soft nproc 8192
tidb hard nproc 8192
EOF'
done
```

跑完后，每台 VM 应看到以下输出：
```
=== DONE on <hostname> ===
swap_kb=0
thp=always madvise [never]
chrony=active
tidb_uid=...（数字）
```

## 步骤 3：把控制机公钥推到三台 VM

> 用 `tidb` 用户登录的免密通道。预计 3 min。

```bash
for ip in 192.168.56.11 192.168.56.12 192.168.56.13; do
  ssh-copy-id -i ~/.ssh/id_rsa_tidb.pub tidb@$ip
done

# 验证：不再要密码
for ip in 192.168.56.11 192.168.56.12 192.168.56.13; do
  ssh -i ~/.ssh/id_rsa_tidb tidb@$ip 'hostname && id -u tidb'
done
```

如果 root 用 prep 脚本建了 `tidb` 用户但没设密码，`ssh-copy-id` 会失败 —— 这时手动把公钥追到每台的 `/home/tidb/.ssh/authorized_keys`：

```bash
PUB=$(cat ~/.ssh/id_rsa_tidb.pub)
for ip in 192.168.56.11 192.168.56.12 192.168.56.13; do
  ssh root@$ip "echo '$PUB' >> /home/tidb/.ssh/authorized_keys && chown tidb:tidb /home/tidb/.ssh/authorized_keys && chmod 600 /home/tidb/.ssh/authorized_keys"
done
```

## 步骤 4：`tiup cluster check`（对应 EX9 / KP91）

```bash
cd /Users/zhaoyueshun/project/tidb_info/install
tiup cluster check ./tidb-topology.yaml -u tidb -i ~/.ssh/id_rsa_tidb
```

会输出每台的检查项：CPU 频率、内存、磁盘、内核参数、THP、time-sync 等。在 1GB PoC 上预期会看到这些 **WARN 但不致命**：

| 项 | 现象 | 处理 |
| --- | --- | --- |
| memory | < 推荐 32GB | PoC 无视 |
| cpu | < 推荐 8 核 | PoC 无视 |
| disk-performance | fio 读写延迟 fail | PoC 无视 |
| selinux | （Debian）报错 | prep 脚本已经放了 setenforce stub |

**如果有 FAIL（红色）**，先看清楚是什么再决定。**THP / swap / time-sync** 失败是必须先解决的，其它的 warn 可以 `--apply` 试着修，或者忽略。

> **学习点对照 KP91**：`tiup cluster check` 本质是把一套 shell 探针 push 到目标机执行，你可以打开 `~/.tiup/storage/cluster/clusters/<name>/audit/` 看到完整日志。

## 步骤 5：`tiup cluster deploy`（对应 EX3 / EX4 / KP86）

```bash
tiup cluster deploy mytidb v7.5.1 ./tidb-topology.yaml \
  -u tidb -i ~/.ssh/id_rsa_tidb
```

会跳出确认提示，看清楚要部署哪些实例后输入 `y`。

部署过程中 TiUP 会：
1. 在三台 VM 上建 `/tidb-deploy` 和 `/tidb-data` 目录
2. 下载对应版本的二进制 + 配置渲染到 `/tidb-deploy/<component-port>/bin/` 和 `/conf/`
3. 注册 systemd unit（`*-pd-2379.service` 等）

部署完成后看本地：
```bash
ls ~/.tiup/storage/cluster/clusters/mytidb/
# meta.yaml + ssh / audit / config 目录
cat ~/.tiup/storage/cluster/clusters/mytidb/meta.yaml | head -40
```

> **学习点对照 KP77**：理解 `meta.yaml` 是 TiUP 的"事实表"。后续 scale-out / upgrade / edit-config 都先改 meta，再下发。任何手动改远程机器的配置都会在下次 `reload` 时被 meta 覆盖。

## 步骤 9：故障注入小实验（对应 EX5 / KP11 / KP19）

> **这一步是阶段 1 最有学习价值的部分**，不要跳过。

### 实验 9a：杀掉 debian3 的 TiKV，观察 Region 迁移

```bash
# 控制机
ssh -i ~/.ssh/id_rsa_tidb tidb@192.168.56.13 'sudo systemctl stop tikv-20160.service'

# 在另一个终端，每 5s 看一次 store 状态
tiup ctl:v7.5.1 pd -u http://192.168.56.11:2379 store
```

观察要点：
1. 多久之后 store 3 从 `Up` 变 `Disconnected` → `Down`（默认 30 分钟后变 Down，但你设了非默认的 raft 心跳，恢复决策也会偏慢）
2. Leader count 怎么变（其它 store 的 leader 数会涨）
3. Region count 怎么变（其它 store 不会立刻补副本，因为还在 `Disconnected` 阶段 PD 不会调度）

恢复：
```bash
ssh -i ~/.ssh/id_rsa_tidb tidb@192.168.56.13 'sudo systemctl start tikv-20160.service'
```

### 实验 9b：PD Leader 主动切换（对应 EX23 / KP7）

```bash
tiup ctl:v7.5.1 pd -u http://192.168.56.11:2379 member leader
# 假设当前 leader 是 pd-1
tiup ctl:v7.5.1 pd -u http://192.168.56.11:2379 member leader resign

# 等 3s
tiup ctl:v7.5.1 pd -u http://192.168.56.11:2379 member leader
# 应该已经换人
```

观察 TiDB 一侧（在 mysql 客户端开着）：
- `SELECT NOW(), TIDB_VERSION();` 会不会卡几秒
- TiDB log（debian1 / debian2）有没有 PD leader change 的提示

## 常见踩坑（看到再回来翻）

### 1. `tiup cluster check` 报 `Get \"http://...:9100\": ...`
→ Node exporter 端口还没启动，正常现象，`check` 不会真的去连未部署的节点。

### 2. `Error: failed to scp...`
→ 99% 是 SSH key 没正确推到目标机的 tidb 用户。回步骤 3。

### 3. start 之后某个 TiKV 状态 Down，logs 里写 `cluster id mismatch`
→ 之前部署过同集群名的残留数据没清。`tiup cluster destroy mytidb` 再重新 deploy。**对应 DS32**。

### 4. start 之后 Grafana 打不开
→ Debian arm64 上 Grafana 偶尔 systemd 启动慢，`ssh debian1 systemctl status grafana-3000` 看。

### 5. debian1 上 TiKV 启动后秒被 kill
→ OOM。看 `dmesg | grep -i kill` 确认。回审计 A，把 Grafana 挪走或扩内存。

### 6. mysql 连接 TiDB 报 `1130 - host is not allowed`
→ 罕见，是 root 账号默认绑了 `%` host，理论不该出现。检查是不是 `--init` 之后改过 root 的 host 字段。
